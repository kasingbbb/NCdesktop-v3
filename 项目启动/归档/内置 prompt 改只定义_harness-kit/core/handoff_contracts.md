# 角色间交接契约（Handoff Contracts）

> **设计原则**：Agent 之间不依赖长对话上下文传递信息。所有交接通过结构化文档完成，每份文档有明确的必含字段。交出的文档如果缺少必含字段，**接收方有权拒收并要求补全**。

---

## 1. Debate → Conductor（PRD 桥接契约）

### 触发条件
Debate 机制完成（四层共识产出完毕），正式 PRD 已写入。

### 交付物
正式 PRD 文档末尾必须附加一份 **Conductor 可读摘要**：

```markdown
## Conductor 桥接摘要

### 核心功能清单（带优先级）
| 功能 | 优先级 | 核心用户场景 | 来自 Debate 的关键约束 |
|------|--------|-------------|----------------------|
| [功能名] | P0/P1/P2 | [一句话] | [Debate 中确认的约束] |

### 不可妥协的技术底线
1. [从 Debate 共识中提取的硬约束]
2. ...

### 已识别的高风险项
| 风险 | 来源（Debate 哪一轮） | 当前状态 | 缓解策略 |
|------|---------------------|----------|----------|
| [风险描述] | Round N | 已解决/已搁置/待定 | [策略] |

### MVP 边界声明
- **做什么**：[明确列表]
- **不做什么**：[明确列表，附原因]

### Debate 中未达成共识的争议
[如有，列出。Architect 在设计时必须对这些争议做出明确选择。]
```

### 接收方检查
Conductor 在启动 ARCHITECTURE 状态前，必须验证：
- [ ] 核心功能清单存在且每项有优先级
- [ ] MVP 边界声明的"不做什么"非空
- [ ] 高风险项已标注状态

---

## 2. Architect → Dev（Task 输入契约）

### 触发条件
Architect 完成技术方案，task 清单已写入 `progress.md`。

### 交付物
每个 Task 的 `input.md` 必须包含以下字段：

```markdown
# Task 输入 — task_00N_xxx

## 目标
[一句话说清楚这个 task 做什么]

## 前置条件
- 依赖 task：[task_ID 列表，或"无"]
- 必须先存在的文件/接口：[列表]

## 验收标准（Acceptance Criteria）
1. [AC-1：可验证的完成条件]
2. [AC-2：可验证的完成条件]
3. ...

## 技术约束
- [从 session_context.md 的代码规范中提取的相关约束]
- [Architect 方案中对该 task 的特定约束]

## 参考文件
- [需要阅读的相关已有代码路径]
- [Architect output.md 中与该 task 相关的段落引用]

## 预估影响范围
- 新建文件：[列表]
- 修改文件：[列表]
```

### 接收方检查
Dev 开始工作前，必须验证：
- [ ] 目标描述清晰，一句话可概括
- [ ] 验收标准可测试/可验证
- [ ] 前置条件的 task 已处于 DONE 状态

---

## 3. Dev → Reviewer（交付契约）

### 触发条件
Dev 完成代码实现，所有测试通过。

### 交付物
`output.md` 必须包含以下字段：

```markdown
# Task 交付 — task_00N_xxx

## 实现摘要
[简要描述做了什么，核心设计决策]

## 修改的文件
| 文件路径 | 变更类型 | 说明 |
|---------|----------|------|
| `path/to/file` | 新建/修改/删除 | [一句话] |

## 对 Architect 方案的遵守声明
- [ ] 目录结构与 Architect 方案一致
- [ ] API 路径/命名与 Architect 方案一致
- [ ] 数据模型与 Architect 方案一致
- [ ] 未引入计划外的新依赖
- 偏离说明（如有）：[具体偏离项和原因]

## 测试命令
```bash
[精确的测试运行命令]
```

## 测试结果
```
[实际运行输出，完整粘贴]
```

## 自测验证矩阵
| 场景类型 | 场景描述 | 状态 | 结果/原因 |
|----------|----------|------|-----------|
| ✅ 正常路径 | [描述] | 已测 | [PASS/FAIL + 说明] |
| ⚠️ 边界条件 | [描述] | 已测/未测 | [结果或跳过原因] |
| ❌ 异常路径 | [描述] | 已测/未测 | [结果或跳过原因] |

## 已知局限
[自己意识到但未修复的问题，附原因]

## 需要 Reviewer 特别关注的地方
[具体位置和关注理由]
```

### 接收方检查
Reviewer 在开始审查前，必须验证：
- [ ] 测试结果存在且非空
- [ ] 自测验证矩阵存在且正常路径全部 PASS
- [ ] 架构遵守声明已填写

---

## 4. Reviewer → Dev（Fix 指引契约）

### 触发条件
Reviewer 判断为 FIX（非 PASS、非 BLOCKER）。

### 交付物
`review_scorecard.md` 中的 **修复指引** 节必须包含：

```markdown
## 给 Dev 的修复指引

### 问题清单（按优先级排序）

#### BLOCKER（必须修复，否则不可能 PASS）
1. **问题**：[具体描述]
   - **代码位置**：[文件:行号]
   - **修复方向**：[建议的修复思路]
   - **验证标准**：[怎样才算修好了]

#### MAJOR（强烈建议修复）
1. 同上格式...

#### MINOR（可选修复）
1. 同上格式...

### 修复范围约束
- **只修以上列出的问题**，不要连带重构
- **如果修复涉及已通过 task 的代码，必须在交付中标注**
- 修复完成后，必须重跑所有原有测试 + 针对修复项的新测试
```

### 接收方检查
Dev 在开始修复前，必须验证：
- [ ] 每个问题都有验证标准
- [ ] 修复范围约束已明确

---

## 5. Conductor ↔ 所有 Worker（状态传递契约）

### progress.md 最小必含字段

```markdown
# Conductor Progress

## 当前状态
STATE: [状态值]
当前 Task: [task_id]
更新时间: [timestamp]

## 已完成 Tasks
[列表]

## 当前 Task 详情
Task ID:
描述:
状态:
交付物路径:

## 待执行 Task 队列
[列表]

## 已知问题 / Blockers
[列表或"无"]

## 关键决策记录
[时间戳 + 决策内容]

## 状态转移日志
[timestamp] STATE: X → Y | 原因: [一句话] | 风险: [无/低/中/高]
```

### 规则
- **每次状态变更后，Conductor 必须立即更新 progress.md**
- **新 Agent 启动时，第一步必须读取 progress.md + session_context.md**
- **progress.md 是唯一的状态真相来源，不得依赖对话上下文**
