## 1. 项目概述

### 1.1 背景

NCdesktop 通过 LLM 实现四个核心知识处理功能，每个功能对应一条内置 Prompt：

| # | 功能模块 | Prompt 作用 | 当前状态 |
|---|---------|------------|---------|
| 1 | 文件打标签 | 输入文件内容，输出标签列表 | 内置，不可调整 |
| 2 | PARA 分组 | 输入文件内容+元信息，输出 P/A/R/A 分类归属 | 内置，不可调整 |
| 3 | 知识概念提取 | 输入文件内容，输出核心概念及定义/关系 | 内置，不可调整 |
| 4 | 知识聚合 | 输入多个知识碎片，输出结构化聚合知识 | 内置，不可调整 |

### 1.2 核心问题

部分用户（专家型用户）有自己的打标签习惯、分组习惯、知识概念提取偏好，内置 Prompt 无法满足其个性化需求。

### 1.3 解决方案

在设置中新增「Prompt 自定义」区域，将四条内置 Prompt 暴露为可编辑的文本，专家用户可直接修改 Prompt 文本。普通用户无需关注，默认使用内置 Prompt 即可。

---

## 2. 用户定义

### 2.1 目标用户

**专家用户**：理解 Prompt 工程，有能力编写和调优 LLM Prompt 的用户。这是一个高级功能，不面向普通用户。

### 2.2 核心场景

> 一个有 Prompt 编写经验的用户，觉得内置的打标签 Prompt 生成的标签粒度不够细，或者风格不符合自己的知识管理体系。他打开设置 → Prompt 自定义 → 文件打标签，看到当前的 Prompt 文本，直接修改为自己写的版本，保存。之后系统用他的 Prompt 来打标签。如果效果不好，随时恢复默认。

### 3.2 用户可编辑的 Prompt（共 4 条）

| # | Prompt 名称 | 展示标题 | 说明 |
|---|------------|---------|------|
| 1 | tagging_prompt | 文件打标签 | 控制 LLM 如何为文件生成标签 |
| 2 | para_grouping_prompt | PARA 分组 | 控制 LLM 如何将文件归类到 P/A/R/A |
| 3 | concept_extraction_prompt | 知识概念提取 | 控制 LLM 如何从内容中提取概念 |
| 4 | knowledge_aggregation_prompt | 知识聚合 | 控制 LLM 如何将多个碎片聚合为结构化知识 |

### 3.3 功能行为

| 行为 | 说明 |
|------|------|
| 默认值 | 每条 Prompt 的默认值 = 当前内置 Prompt 文本 |
| 编辑 | 用户在文本编辑框中直接修改 Prompt 内容 |
| 保存 | 点击保存后，后续该功能的 LLM 调用使用用户自定义版本 |
| 恢复默认 | 单条恢复：将该 Prompt 恢复为内置默认值；全部恢复：四条全部恢复 |
| 状态指示 | 已自定义的 Prompt 显示「● 已自定义」标记，未修改的显示「● 默认」 |
| 持久化 | 用户自定义的 Prompt 保存到本地 SQLite，重启后仍然生效 |

## 4. 数据模型

```sql
CREATE TABLE user_custom_prompt (
  module TEXT PRIMARY KEY,    -- 'tagging' | 'para' | 'concept' | 'aggregation'
  prompt_text TEXT NOT NULL,  -- 用户自定义的 Prompt 全文
  is_custom BOOLEAN DEFAULT 0,
  updated_at TIMESTAMP
);
```

运行时逻辑：
1. 查询 `user_custom_prompt` 表中对应 module 的记录
2. 如果 `is_custom = 1`，使用 `prompt_text` 作为 LLM 的 system prompt
3. 如果 `is_custom = 0` 或记录不存在，使用代码中的内置默认 Prompt

## 5. 非功能需求

| 维度 | 要求 |
|------|------|
| 安全 | 用户自定义 Prompt 仅存储在本地 SQLite，不上传云端 |
| 安全 | 用户 Prompt 替换的是 system prompt 中的业务指令部分，不影响输出格式约束 |
| 性能 | 配置加载 < 100ms，不影响 LLM 调用流程 |
| 可用性 | 一键恢复默认，降低试错成本 |
| 兼容性 | 内置 Prompt 升级时，已自定义的用户不受影响（继续使用自己的版本） |

## 6. 技术约束

| 约束 | 说明 |
|------|------|
| Tauri 架构 | 前端 TS 负责编辑框 UI，后端 Rust 负责 SQLite 读写 |
| 本地优先 | 自定义数据不离开本机 |
| Token 上限 | 用户自定义 Prompt 不应超出 LLM 上下文窗口的合理范围 |

## Conductor 桥接摘要

### 核心功能清单（带优先级）

| 功能 | 优先级 | 核心用户场景 | 关键约束 |
|------|--------|-------------|---------|
| 设置页 Prompt 自定义区域 | P0 | 专家用户自定义四条 Prompt | 仅在设置页体现 |
| 四条 Prompt 的文本编辑与保存 | P0 | 直接编辑 Prompt 文本 | 本地 SQLite 持久化 |
| 恢复默认功能 | P0 | 试错后一键回退 | 单条恢复 + 全部恢复 |
| LLM 调用时读取自定义 Prompt | P0 | 自定义生效 | 有自定义用自定义，否则用内置 |

### 不可妥协的技术底线

1. 自定义 Prompt 本地存储，不上传云端
2. 一键恢复默认，任何时候可回退
3. 内置 Prompt 升级时不覆盖用户已有的自定义

### 已识别的高风险项

| 风险 | 缓解策略 |
|------|----------|
| 用户写的 Prompt 导致 LLM 输出格式异常，下游功能报错 | 保留输出格式约束作为独立校验层，不受用户 Prompt 影响 |
| 用户 Prompt 过长超出 context window | 保存时检查 token 长度，超限提示 |
| 内置 Prompt 升级后用户自定义版本落后 | 用户自定义优先，不强制覆盖；可通过恢复默认获取最新版本 |
