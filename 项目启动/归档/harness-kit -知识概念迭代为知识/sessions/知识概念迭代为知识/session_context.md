# Session Context — 知识概念迭代为知识

> Session 创建时间: 2026-04-11
> 复杂度等级: L（功能核心重构，涉及用户认知模型，用户可见 UI 是产品核心）

---

## 1. 项目信息 [必填]

- **项目名称**：NCdesktop（NoteCapt）— 知识功能迭代
- **一句话描述**：将 NCdesktop 的「知识关联」功能，从"概念名称+一句话定义的引用层"进化为"真正的知识介绍与解释层"，让用户能够系统性地了解、吸收、成长、进化和引用知识。
- **项目类型**：Desktop App（Tauri + React）
- **复杂度等级**：L

---

## 2. 技术上下文 [必填]

- **主语言**：TypeScript（前端）+ Rust（后端 Tauri commands）
- **框架/运行时**：Tauri 2.x + React 18 + Zustand（状态管理）+ SQLite（本地存储）
- **数据库**：SQLite（本地，通过 Tauri 命令访问）
- **关键外部依赖**：LLM API（用户自配 endpoint，支持 OpenAI 兼容接口）；已有 llmProbe 和 llmPreview Tauri commands
- **现有代码库**：改造现有代码（知识关联功能已部分实现，见 v2.1 PRD）
- **目标部署环境**：本地桌面端（macOS 优先，Windows 兼容）

---

## 3. 关键约束 [必填]

- **安全性要求**：高 — 所有数据本地存储，LLM 调用走用户自配 API，不上传用户文档到任何第三方服务
- **性能要求**：高 — 知识页面首屏 ≤ 500ms，LLM 生成首字 ≤ 3s，知识检索 ≤ 100ms
- **用户体验要求**：高 — 这是产品的核心差异化功能，用户体验是第一优先级
- **可维护性要求**：中 — 一人代码（One-man Code）实践，代码要清晰可读

- **不可妥协的底线**：
  1. 所有用户文档内容不得上传到任何第三方服务（隐私红线）
  2. LLM 生成内容必须有明确的来源追溯，不得脱离用户自己的文档凭空生成
  3. 用户手动编辑的内容（定义、笔记）必须持久化，不得被 AI 重新生成覆盖
  4. 功能必须在无网络环境下可访问已生成内容（离线查看）

## 5. 领域特定代码规范

```
- TypeScript：严格类型，避免 any；接口定义放在 types/ 目录
- Zustand Store：不 import 其他 Store，跨 Store 数据在组件层组合
- Tauri Commands：命名规范 snake_case，入参/出参有 Rust 结构体定义
- LLM 调用：流式输出优先；Prompt 模板集中管理，不散落在组件中
- SQLite：所有写操作在 Rust 侧执行，前端只读取
- 错误处理：只在系统边界（LLM API、文件 IO）处理异常，内部逻辑不防御
```

## 10. 当前现状（Debate 输入背景）

### 已实现的 v2.1 知识关联功能
1. **概念提取**：从文档中提取学术概念（名称 + 别名 + 一句话定义 + 原文摘录）
2. **观点聚合**：同一概念在不同课程/文档中的不同解读视角
3. **案例引用**：文档中的具体引文，可跳转原文
4. **知识拓展**：前置知识（upstream）和应用方向（downstream）— 只读，静态展示

### 核心问题（用户指出）
当前设计的本质是「概念词汇表」（Glossary）：
- 只告诉用户"这个概念是什么"（一句话定义）
- 没有帮助用户"真正理解这个概念"
- 知识获取 → 知识进化 → 知识成长的递进路径缺失
- 用户不能从「知道概念存在」进化到「能运用这个概念」

### 期望的进化方向
从「知识概念的介绍和引用」进化为「真正的知识介绍与解释」：
1. 针对某个文档或某些类型文档的知识
2. 针对某个方向的文档的知识
3. 针对某个文件夹内文档的知识
4. 未来：结合需求形成 skill 或工作流技能的输出
