## 验收标准（Acceptance Criteria）

本 task 的产出是一份**审查报告**（写入 task_010 的 output.md），包含以下所有检查项的结果：

### A. 端到端流程验证（必须全部通过）

1. **AC-A1**（Feature Discovery）：全新状态下打开知识关联视图，Tooltip 正确显示；第二次进入不再显示。
2. **AC-A2**（入口按钮）：所有已有概念的详情页均有「深入理解」蓝色按钮，视觉上与其他辅助按钮有明显区分。
3. **AC-A3**（空状态引导）：未触发过深入理解的概念，Definition 区域下方有「想深入理解这个概念？」弱引导文字。
4. **AC-A4**（深入理解页面）：点击「深入理解」后正确进入深入理解页面；透明度声明横幅可见。
5. **AC-A5**（文档整合摘要）：首次打开（无缓存）时流式加载，第二次打开（有缓存）≤ 500ms 内显示内容。
6. **AC-A6**（来源链接）：摘要区域的每个来源文档名有标注；「展开原文」功能可用。
7. **AC-A7**（理解框架）：点击「生成理解框架」后流式加载，每个条目有来源标注；「查看依据」展示原文段落。
8. **AC-A8**（重新生成）：「重新生成」按钮正确触发新的 LLM 调用，新内容替换旧内容，不出现新旧内容混合。
9. **AC-A9**（用户笔记）：在 textarea 输入后停止 1s，「已保存」提示出现；重新打开页面后笔记内容恢复。
10. **AC-A10**（镜子反馈）：点击「和 AI 核对一下」后，反馈内容符合探索式措辞（无批改式语言）；来源标注存在。
11. **AC-A11**（概念关系网络）：关联概念卡片正确显示，文字为「一起出现在 [文档名]」；空状态时显示正确提示。
12. **AC-A12**（关联概念导航）：点击关联概念名称，正确导航到该概念详情页。
13. **AC-A13**（返回导航）：「← 返回概念列表」可用，点击后返回到正确状态。

### B. 措辞审查（Reviewer 重点检查项）

14. **AC-B1**：镜子反馈文本中不出现以下词语：「不完整」「有误」「遗漏了」「错误」「不正确」「你缺少」「你的理解有问题」（对多个不同概念测试 3-5 次，记录所有实际输出文本）。
15. **AC-B2**：关系网络区域不出现「紧密相关」「深层联系」「强关联」等词。
16. **AC-B3**：透明度声明横幅文字与 PRD 规范一致（完整核对文字）。
17. **AC-B4**：理解框架各模块标题（核心机制、典型场景、常见误区、一句话精华）与 PRD 一致。

### C. 安全底线验证

18. **AC-C1**：`concept_user_notes.user_explanation` 中存储的内容与 `concepts.definition` 字段完全独立——通过 SQLite CLI 直接查询两张表验证，修改一个不影响另一个。
19. **AC-C2**：`user_edited = true` 的概念定义未被任何新功能修改（可手动在数据库将一个 concept 的 `user_edited` 设为 true，再执行各种新功能操作，验证定义不变）。
20. **AC-C3**：LLM 调用时发送的 Prompt 内容确认只包含用户文档的摘录（通过查看 Rust 侧 `prompts.rs` 的实际 Prompt 输出，或添加临时 `eprintln!` 日志验证）。

### D. 性能验证（有缓存时）

21. **AC-D1**：有 summary 缓存时，深入理解首屏内容可见时间 ≤ 500ms（可使用 DevTools 的 Performance 或 Network 面板计时）。
22. **AC-D2**：概念关系网络从页面挂载到内容可见 ≤ 100ms（同上）。
23. **AC-D3**：自动保存不阻塞 UI（输入时页面不卡顿）。

### E. 回归验证（v2.1 功能不受影响）

24. **AC-E1**：v2.1 概念列表、观点聚合、案例引用、知识拓展功能在此次迭代后仍正常工作。
25. **AC-E2**：v2.1 已有数据（concepts、concept_viewpoints、concept_cases、concept_extensions 表）无任何数据丢失或字段变更（通过 SQLite CLI 对比 migration 前后的 `.schema` 输出）。

## 技术约束

- **本 task 不做代码修改**：UX 审查的职责是发现问题并记录，不是在此 task 中直接修改代码；若发现 BLOCKER 级问题，在报告中标注，等待对应 Dev task 修复
- **报告格式**：输出写入 `task_010_ux_review/output.md`，格式参考 `handoff_contracts.md` 中的 Dev → Reviewer 交付格式（调整为 UX Reviewer 视角）；每个 AC 标注 PASS / FAIL / PARTIAL + 说明
- **测试数据**：需要数据库中有至少 2-3 个已提取过概念的文档（具有 `concept_cases.excerpt` 数据），才能完整测试所有功能；若无测试数据，报告中说明哪些 AC 因数据不足跳过，给出复测建议
