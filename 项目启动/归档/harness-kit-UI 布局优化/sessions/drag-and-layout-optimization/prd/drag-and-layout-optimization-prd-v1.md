# PRD — NCdesktop 拖拽外发与中栏布局优化

> 版本: v2.0 | 产出日期: 2026-04-12
> 来源: Debate Session + PM 核心需求修正
> 复杂度等级: M（中等）

---

## 1. 项目概述

### 1.1 核心问题（PM 原话）

> 当前的文件管理功能承载的目的是呈现、点击查看、关联"打标后的文档"的作用，这些功能是很好很重要的。但是还不能更好的进行"拖拽发送、拖拽多选、拖拽发送到外部应用、网页"的功能，只有这样才能够**一键选中转化后的各种模态的素材，一次过丢给外部工具进行 AI 消化使用**。

### 1.2 两大核心迭代目标

| # | 目标 | 一句话定义 |
|---|------|-----------|
| **核心 1** | **进得来、出得去** | 框选多个素材 → 一次拖出到 Finder / 微信 / Keynote / 邮件等外部应用，实现「一键抓取、一次投喂」的操作闭环 |
| **核心 2** | **看得多、选得快** | 中栏列表项/网格项视觉紧凑化，像调分辨率一样让一页显示更多文件，支持快速浏览和批量选取 |

### 1.3 当前能力盘点

**已有且保留的好功能：**
- ✅ 双栏呈现（导入原件 + 工作区）
- ✅ AI 标签与归类目录展示
- ✅ 拖入导入（DropzoneApp + import_drop_paths）
- ✅ Inspector 详情面板
- ✅ 标签筛选 + 工作区文件夹筛选
- ✅ 文档查看器（DocumentViewer）

**缺失的核心能力（本次迭代要补上的）：**
- ❌ 拖出到外部应用（当前 dataTransfer 仅设置自定义 MIME，Finder 无法识别）
- ❌ 一屏可见文件数太少（列表项高约 80px，14 寸屏一屏仅约 9 个）
- ❌ 网格缩略图仅图标，无法视觉识别照片

**Debate 中发现的存量 Bug（顺带修复）：**
- 🐛 MIME 不匹配：`useDragAssets`（`application/notecapt-assets`）与 `useKeyframeDrop`（`application/x-asset-id`）不一致，拖到时间轴不工作
- 🐛 框选不可用：素材卡片用 `<button>` 包裹，`useRubberBandSelect` 排除了 `<button>`

## 2. 用户画像与核心场景

### 2.1 核心用户

学生/知识工作者，课后整理完素材后，需要**批量选取 → 一次投喂给 AI 工具或分享给同学**。

### 2.2 核心场景（按优先级排序）

**场景 A（最高优先级）：批量外发给 AI 工具**
```
用户在 NCdesktop 工作区中框选/Cmd+Click 选中 5 个素材
  （2 张照片 + 1 个 PDF + 2 个 Markdown 文件）
→ 直接拖拽到桌面上的 Finder 文件夹
→ 5 个文件出现在 Finder 中
→ 用户从 Finder 将文件上传到 ChatGPT / NotebookLM / Keynote
```

**场景 B（高优先级）：直接拖到微信/邮件/Keynote**
```
用户选中 3 张课堂照片
→ 直接从 NCdesktop 拖到微信对话窗口
→ 3 张照片作为附件发送
```

**场景 C（高优先级）：快速浏览选取**
```
一个项目有 50 个素材
→ 用户切到紧凑列表视图，一屏看到 20+ 个文件
→ 快速扫一眼，Cmd+Click 选中需要的 8 个
→ 拖出
```

## 3. 功能需求

### 3.1 核心功能 1：拖拽外发到外部应用

#### F01: 系统级文件拖出（核心中的核心）

| 字段 | 值 |
|------|-----|
| 优先级 | **P0** |
| 估时 | 需 Spike 验证后确定（Spike 本身 0.5d） |
| 影响文件 | `src/hooks/useDragAssets.ts`、`src-tauri/Cargo.toml`、`src-tauri/src/lib.rs` |

**需求描述**：用户在工作区选中 1 个或多个素材后，可以直接拖拽到 macOS Finder、微信、Keynote、邮件等外部应用，接收方得到的是**真实文件**。

**技术方案（需 Spike 验证优先级）**：

| 方案 | 原理 | 优先验证 |
|------|------|---------|
| **方案 A: `tauri-plugin-drag`** | CrabNebula 官方插件，底层封装 NSPasteboard + beginDraggingSession | ⭐ 首选 |
| **方案 B: Rust 侧自研 `start_drag` 命令** | 通过 `objc2` crate 直接操作 NSPasteboard | 备选（如 A 不兼容） |
| **方案 C: `text/uri-list` + `file://`** | 纯 Web API 在 dataTransfer 中写入文件 URI | 最简但可能被 WKWebView 限制 |

**Spike 验证计划（0.5 天）**：
1. 引入 `tauri-plugin-drag` 依赖
2. 在一个测试按钮上调用 `startDrag({ item: ["/path/to/test.pdf"] })`
3. 验证：拖到 Finder 是否产生文件？拖到微信是否能发送？
4. 验证多文件（5 个以上）性能
5. 如果方案 A 失败，立即转 B 或 C

**交互规格**：
- 选中 1 个素材 → 拖拽 → 拖出窗口 → Finder 收到文件
- 选中 N 个素材 → 拖拽 → 拖出窗口 → Finder 收到 N 个文件
- 拖拽过程中显示**文件数量徽章**（蓝色圆形 badge，显示 `N`）
- 拖拽中原位卡片 `opacity: 0.4` 半透明
- 拖出的是文件**引用**（不复制，指向 `asset.filePath` 本地路径），外部应用按需读取

**关键约束**：
- 传递的是 `asset.filePath`（工作区内的真实文件路径），不修改原件
- 必须支持多选拖出（框选 / Cmd+Click 选中的全部）
- 不阻塞 UI（大文件应异步处理）

**验收标准**：
- [ ] 单个素材可拖到 Finder，Finder 中出现对应文件
- [ ] 多选 5 个素材可拖到 Finder，5 个文件全部出现
- [ ] 拖到微信聊天窗口，文件作为附件可发送
- [ ] 拖出过程中显示数量徽章
- [ ] 100ms 内启动拖拽（无感知延迟）

#### F02: 框选修复 + 手势路由

| 字段 | 值 |
|------|-----|
| 优先级 | **P0** |
| 估时 | 2h |
| 影响文件 | `src/hooks/useRubberBandSelect.ts`、`src/hooks/useDragAssets.ts`、`src/components/features/AssetListView.tsx` |

**需求描述**：修复当前框选不可用的 Bug，并建立框选 vs 拖拽的手势路由。

**Bug 根因**：
- 素材卡片用 `<button>` 包裹 → `useRubberBandSelect` 的 `target.closest("button")` 排除 → 框选无法触发
- 只能从 8px 的 `gap-2` 间隙开始框选 → 实际不可用

**修复方案**：
1. 素材卡片从 `<button>` 改为 `<div role="button" tabIndex={0} data-asset-card>`
2. `useRubberBandSelect` 排除条件改为 `target.closest("[data-no-rubber]")`
3. 引入 5px 移动阈值区分「点击选中」和「框选/拖拽」

**手势路由规则**：

| 起点 | 动作 | 行为 |
|------|------|------|
| 空白区域 mousedown | mousemove > 5px | **框选** |
| 未选中卡片 mousedown | mouseup（无移动） | **选中该卡片** |
| 未选中卡片 mousedown | mousemove > 5px | **先选中，再启动拖拽** |
| 已选中卡片 mousedown | mousemove > 5px | **拖拽全部已选** |
| 已选中卡片 mousedown | mouseup（无移动） | **缩减为只选该卡片** |
| Cmd + mousedown | mouseup | **toggle 选中/取消** |

**验收标准**：
- [ ] 空白区域拖动 → 出现框选矩形
- [ ] 素材卡片上方拖动 → 框选（非拖拽）
- [ ] 已选素材上拖动 → 拖拽（非框选）
- [ ] Cmd+Click → 加入/移除选中集
- [ ] 框选矩形视觉：`rgba(31,69,110,0.08)` 底色 + `rgba(31,69,110,0.3)` 边框

### 3.2 核心功能 2：UI 紧凑化（「看得多、选得快」）

#### F05: 列表视图紧凑化

| 字段 | 值 |
|------|-----|
| 优先级 | **P0** |
| 估时 | 2h |
| 影响文件 | `src/components/features/AssetListView.tsx` |

**需求描述**：大幅缩小列表项高度，让一页能看到更多文件。当前列表项约 80px 高（一屏 ~9 个），目标缩到 36-40px（一屏 ~18-20 个）。

**当前布局 vs 目标布局**：

```
当前（~80px 每项）：
┌────────────────────────────────────────────────────────┐
│ [icon 18px]  文件名（最多两行 line-clamp-2）           │
│              [标签1] [标签2] [标签3]                    │  ← 80px
│              类型 · 大小                               │
└────────────────────────────────────────────────────────┘
  gap: 8px

目标（~36px 每项）：
┌────────────────────────────────────────────────────────┐
│ [icon 16px]  文件名（单行 truncate）    [类型]  1.2MB  │  ← 36px
├────────────────────────────────────────────────────────┤  ← 1px divider
│ [icon 16px]  另一个文件名               [PDF]   340KB │  ← 36px
└────────────────────────────────────────────────────────┘
```

**具体参数变化**：

| 属性 | 当前 | 目标 |
|------|------|------|
| 行高 | ~80px（py-2.5 + 3 行文本 + gap-2） | **36px**（py-1 + 单行 + 1px divider） |
| 内边距 | `px-3 py-2.5` | **`px-2 py-1`** |
| 间距 | `gap-2`（8px） | **`gap-0`**（用 1px divider 替代） |
| 图标 | 18px | **16px** |
| 文件名 | `line-clamp-2`（两行截断） | **`truncate`（单行截断）** |
| 标签行 | 显示（占 ~20px） | **不显示**（由 Sidebar 标签筛选承接） |
| 元数据行 | 独立一行（类型 · 大小） | **合并到文件名行右侧对齐** |
| 一屏数量 | ~9 个 | **~18-20 个** |
| 字号 | text-sm (14px) | **13px** |
| 文件大小 | 和类型同行 | **右对齐 mono 字体，11px** |

**信息补偿**：
- 标签虽然不在列表中显示，但 hover tooltip 展示完整信息（文件名全称 + 标签 + 路径）
- ExtractionBadge 保持 inline 显示
- 已重命名标记保留（缩小为小圆点）

**一屏数量计算（14 寸 MacBook 1440×900）**：
```
可用高度 ≈ 728px（减去标题栏 + Toolbar + header + padding）
36px + 1px divider = 37px 每项
728 / 37 ≈ 19.7 → 一屏约 19-20 个素材 ✅
```

**验收标准**：
- [ ] 列表项高度 ≤ 40px
- [ ] 14 寸 MacBook 一屏可见 ≥ 18 个素材
- [ ] 文件名 hover 显示 tooltip（完整名称 + 标签）
- [ ] 文件类型和大小在行内右侧可见

### 3.4 不做什么（明确排除）

| 排除项 | 原因 |
|--------|------|
| 密度档位切换（紧凑/标准/宽松手动切换） | 直接默认紧凑即可，不需要用户手动选 |
| 拖出到浏览器网页上传区 | 跨进程限制，浏览器场景用「导出到文件夹」后手动上传 |
| 手势仲裁层完整状态机 | 用 5px 阈值 + 元素判断的最简策略先上线 |
| 虚拟列表 | 当前素材量级下不需要 |

## 4. 技术约束与风险

### 4.1 最大技术风险：系统级文件拖出

Debate 中识别的核心风险是 **WKWebView 的 HTML5 DataTransfer 无法驱动系统文件拖放**。纯 Web `setData("text/uri-list", ...)` 可能被 WKWebView 沙箱限制。

**应对策略：Spike First**

```
Day 0（半天）: Spike 验证
  ├── 测试 tauri-plugin-drag → 能否拖到 Finder？
  ├── 测试 text/uri-list + file:// → WKWebView 是否放行？
  └── 如果都不行 → 自研 Rust 侧 NSPasteboard 命令

Spike 结果决定后续路径：
  ├── 成功 → Day 1-3 正式实现
  └── 失败 → 退化方案：右键「导出到…」+ 「在 Finder 中显示」
```

**退化方案保底**：即使原生拖出 Spike 失败，右键菜单（F04）仍然能提供外部操作能力，用户体验从「一键拖出」退化为「右键→导出」，但功能不缺失。

### 4.2 其他技术约束

| 约束 | 说明 |
|------|------|
| 不破坏拖入导入 | DropzoneApp + import_drop_paths 不动 |
| 不变更数据库 | 无新表 |
| macOS 14+ 兼容 | Apple Silicon 优先 |
| 保持无障碍 | `role="button" tabIndex={0}` |

### 不可妥协的技术底线

1. **拖出必须传递真实文件路径**（不是临时文件）
2. **不破坏拖入导入功能**
3. **框选和拖出不能互相冲突**
4. **紧凑化后文件信息仍可读**（tooltip 补偿）

### 已识别的高风险项

| 风险 | 缓解策略 |
|------|---------|
| tauri-plugin-drag 与当前 Tauri 版本不兼容 | Spike 先行验证；退化到右键菜单 |
| WKWebView 限制 text/uri-list | 如 Web API 不行则走 Rust 原生 NSPasteboard |
| 框选与拖拽手势冲突 | 5px 移动阈值 + 元素类型判断 |
| 紧凑列表中文件名截断不可读 | hover tooltip 显示完整信息 |

### MVP 边界声明

**做什么：**
- 系统级文件拖出（Spike → 实现）
- 列表/网格视图紧凑化
- 框选修复 + MIME 统一
- 右键菜单 + 缩略图 + 视觉反馈

**不做什么：**
- 不做拖出到浏览器上传区（操作系统限制）
- 不做密度切换（直接默认紧凑）
- 不做手势仲裁层完整重构（最简策略先上）
- 不做虚拟列表（当前量级不需要）
