## 修订动机摘要

本文档整合了四轮 Debate 的结论：

1. **Round 1（诊断）**：当前管线翻车样本（"A multi-dimensional evaluation framework..."）暴露三个独立失败：输入污染、primitive 错位、聚类无方向。
2. **Round 2（价值规划）**：定义"知识提取 / 整理 / 记录 / 进化"四阶段的真实价值与介入边界。
3. **Round 3（进化重定义）**：进化触发权从"日历"切到"使用场景"——四种触发模式 A/B/C/D（考试 / 任务 / 阅读 / 回望）。
4. **Round 4（信息架构）**：UI 从"功能列表"切到"模式驱动"——5 模式 + Home + 折叠素材抽屉。

本文档基于这些前提，逐个子功能给出 PRD。

# 第一部分：Extract 阶段（知识提取）

> **阶段价值**：把"被动读过"变成"显式可见的认知 atom"。学生看完一篇文章脑子里有模糊感觉但说不清——提取阶段把这种感受外化成几条具体、有类型、可命名的东西。
>
> **学生第一次拿到的回报**："原来我刚才读到了**这 5 件事**。"

## E1. 文档类型门控

**🎯 Debate**
- Proposer：每份素材都该进抽取管线，过滤会漏掉边缘有用的笔记。
- Reviewer：当前翻车样本（开发宪章被当成"academic concept"）就是没门控的直接证据，**不门控等于污染源被允许进库**。
- Host 决议：必须门控；漏掉的边缘内容由用户后续显式标注追加，远小于污染的代价。

**目标**：阻止非学习材料进入抽取管线，保证库的"信噪比"。

**输入**：
- 新导入的 asset（PDF / MD / 截图 OCR / 文本）
- 文件元数据（来源路径、文件名、来源 App）

**输出**：
- 类型标签 ∈ `{academic, tool_doc, personal_note, reference, other}`
- 置信度 0–1
- 进入下一步流程的 boolean（仅 `academic` 默认通过）

**使用场景**：
1. 学生拖拽混合文件夹（含课件 + 软件文档 + 截图）
2. 系统自动从某监控目录批量导入

**核心交互逻辑**：
1. 导入后 → LLM 一次廉价分类调用（system prompt 简短，截取首 500 字）
2. `academic` → 静默进入 E2
3. `tool_doc / personal_note / other` → 默认归档到"未分类"，**不抽 atom**
4. 学生可在素材列表里手动覆盖类型（"这其实是学术内容，请抽取"）

**介入级别**：🤖（默认零交互）+ 🤝（学生可事后覆盖）

**边界约束**：
- 这一步**不消耗 atom 抽取的 token**，只做分类——成本可控
- 如果文件夹整体被标"学术资料"，可跳过此步，全部走 E2

**优先级**：P0

## E2. Atom 抽取 ⭐

**🎯 Debate**
- Proposer：现行 `concepts` 抽取（词条 + 定义）成熟可靠，加点 prompt 调优就够。
- Reviewer：词条粒度=电子词典，对学生没有思维招式价值。必须切到强类型 atom（5 类）。
- Host 决议：替换 primitive。词条降级为 atom 内部零件，不再作为用户可见单元。

**目标**：从素材中抽出 3–8 条带强类型的 atom，作为知识库的最小可见单位。

**输入**：
- E1 通过的 asset（含 rawText、structuredMd）
- 当前 library 已有的 mental_model 骨架（O1 产物，可空）
- asset 来源元数据

**输出**：每个 atom：
```ts
{
  id: string
  type: 'mental_model' | 'method' | 'claim' | 'exemplar' | 'distinction'
  statement: string         // 一句话主张/规律
  why_it_matters: string    // 为什么这件事值得记
  used_concepts: string[]   // 关联词条名（内部零件）
  source_excerpts: {asset_id, text, offset}[]
  confidence: number
  candidate_skeleton_id: string | null  // O2 反向挂载用
}
```

**使用场景**：
1. 上完一堂经济学课，导入 PPT + 笔记，期望产出"机制 + 方法 + 主张"几条
2. 读完一篇 paper，期望抽出核心 claim + 1–2 个 exemplar

**核心交互逻辑**：
1. E1 通过 → 后台异步触发 LLM 调用
2. Prompt 强制要求每条 atom 必须有 type + statement + why_it_matters + 至少 1 个 source_excerpt
3. 失败/空输出 → 保留为 `unsorted` 状态，不阻断
4. 抽完 → 静默推送 toast：「这篇产出了 5 个新 atom」（不强制点击）

**介入级别**：🤖

**边界约束**：
- 单素材最多产出 8 个 atom（防止 LLM 把目录页也当知识抽）
- 每个 source_excerpt 不超过 200 字，落实 K5 来源原则
- type 不允许为空，无法判定的不写入

**优先级**：P0（**整个系统的根 primitive**）

## E3. Big Idea 标注

**🎯 Debate**
- Proposer：宪章 K2 写了"用户零主动操作"，这一项违反原则。
- Reviewer：K2 在采集元数据上正确，在知识价值判断上错。学生在课后对"big idea"是有强意识的，系统拒绝接收等于浪费最强信号。
- Host 决议：修订 K2 为"默认零交互 + 允许 big idea 强信号"。这一项必须**可跳过**，但要存在。

**目标**：让学生在课后用 10 秒告诉系统"这次最重要的是什么"，加速 atom 抽取的语义聚焦。

**输入**：
- 学生主动触发：素材导入完成后弹窗 / 素材详情页按钮
- 形式：1–2 行文字 OR 30 秒以内语音

**输出**：
- `BigIdeaSignal { asset_id, text, source: 'text'|'voice', timestamp }`
- 作为 E2 抽取 prompt 的额外提示（"重点关注：{big_idea}"）

**使用场景**：
1. 学生上完课立刻拍课件 + 输入"今天讲的核心是泰勒规则的反应函数形式"
2. 读完文章后录音说"作者其实在攻击理性预期假设"

**核心交互逻辑**：
1. 导入完成提示卡片：「想说一句这次最重要的是什么吗？」+ 输入框 + [跳过] 按钮
2. 输入后 → 写入 BigIdeaSignal 表 + 触发 E2 重抽（如已抽则增量补抽）
3. 跳过永不再问（这次素材）；下次新素材重新弹

**介入级别**：🤝（**可跳过**）

**边界约束**：
- 必须做到"跳过"按钮和输入框视觉权重相等，不能引导用户
- 不允许做成必填弹窗
- 不在抽取完成 24 小时后还允许补——这时候记忆已凉

**优先级**：P1

## E4. 课程/学科归属推断

**🎯 Debate**
- Proposer：当前 library 显式归属已经够用。
- Reviewer：学生从一个文件夹拖入混合素材时不会一一指定 library，自动归属是降低门槛的关键。
- Host 决议：默认走自动推断；置信度 < 0.4 时弹二选一。

**目标**：把 asset 自动挂到正确的 library / project，省去手动整理。

**输入**：
- asset 内容 embedding
- 当前所有 library 的"主题向量"（library 内已有 atom 的平均 embedding）
- 时间聚类窗口内其他 asset 的归属

**输出**：
- `inferred_library_id` + `confidence`
- 若 confidence < 0.4：触发二选一确认 UI

**使用场景**：
1. 学生从下载文件夹拖 5 个 PDF 进 App，2 个属"宏观经济学"3 个属"国际金融"
2. 截屏 OCR 得到一段文字，需要自动归库

**核心交互逻辑**：
1. asset 进库 → 计算 embedding → cosine 比对所有 library
2. confidence > 0.7：静默归属
3. 0.4–0.7：右下角小 toast「已归入宏观经济学」（可点击修改）
4. < 0.4：二选一卡片「这是 [宏观经济学] 还是 [国际金融]？」+ 第三选项 [新建 library]

**介入级别**：🤖（高置信）→ 🤝（低置信）

**边界约束**：
- 永远不开放填空，**只允许二选一或三选一**
- 用户手动覆盖一次后，该来源（文件夹/App）永久记忆偏好

**优先级**：P0

## E5. 来源切片绑定

**🎯 Debate**
- Proposer：原文 excerpt 已经在抽取时存了。
- Reviewer：当前 excerpt 是 LLM 输出的字符串，没有 offset，无法跳回原文定位。这违反 K5"可回溯"。
- Host 决议：必须存 offset，可点击跳回原文高亮。

**目标**：每个 atom 绑定回原文的精确位置，回看时一键跳转。

**输入**：
- E2 输出的 source_excerpts 文本
- 原 asset 的 rawText / structuredMd

**输出**：
- 每个 excerpt 增加 `{ offset_start, offset_end, asset_id }`
- 点击 excerpt → 打开原 asset，自动滚动并高亮

**使用场景**：
1. 复习某个 atom 时不确定原文怎么说，一键跳回
2. 写论文引用时需要原文出处

**核心交互逻辑**：
1. E2 抽取后 → 后处理用 fuzzy match 把 excerpt 文本定位回原文
2. 找不到精确位置 → 标记 `offset = -1`，UI 显示"位置未知"但仍可看 excerpt 文本
3. KU/atom 详情页：每条 excerpt 是可点击元素 → 打开 DocumentViewer + 高亮

**介入级别**：🤖

**边界约束**：
- fuzzy match 最少匹配 80%，否则不绑 offset
- 一个 excerpt 在原文出现多次时，绑第一处

**优先级**：P0

## E6. 语音备注转录与分类

**🎯 Debate**
- Proposer：v1.0 §5.2 已设计，沿用即可。
- Reviewer：4 类分类（supplementary/standalone/question/connection）在重新定义后可能变——现在 connection 不再单独成事，并入任务模式更合理。
- Host 决议：保留 4 类，但 connection 类型的后处理改为"在 Home 静默通知里出现"，不强制弹任务。

**目标**：让学生在脑子热的瞬间用语音记下想法，自动归到对的位置。

**输入**：
- 语音文件（来自悬浮录音按钮）
- 录音时间戳
- 时间窗（±10 分钟）内的近期 asset 列表

**输出**：
- `VoiceMemo { id, audioPath, transcript, type, link_target_id, capturedAt }`
- 后处理动作（依 type）：
  - supplementary → 合并到关联 KU 的笔记区
  - standalone → 作为新 atom 候选进 E2
  - question → 在关联 KU 上挂 ❓ 标记
  - connection → Home 通知"你提到了 X 和 Y 可能相关"

**使用场景**：
1. 上课途中按住录音说"老师刚说的这个其实和上节课的需求弹性是一回事"
2. 走路时录"我没听懂为什么 IS 曲线斜率为负"
3. 看完一段视频录"这件事让我想到博弈均衡"

**核心交互逻辑**：
1. 悬浮按钮，按下录音、松开停止（或固定 60 秒上限）
2. 后台 Whisper 转录 → LLM 分类
3. 分类结果以 toast 出现「已归入泰勒规则的笔记区」+ [撤销] [查看]
4. 用户点[撤销]→ 转录文本下降为草稿，存"待处理"队列

**介入级别**：✋（按下录音）→ 🤖（其余）

**边界约束**：
- 转录失败（噪音过大）→ 保留音频，不删除
- 不允许学生"选择类型"——分类是系统的事

**优先级**：P1

# 第二部分：Organize 阶段（知识整理）

> **阶段价值**：把散落的 atom 装进会用的形状——一张能挂住所有素材的 model 地图。
>
> **学生在这阶段拿到的回报**："我懂了哪些事 / 还差哪些事"，**一眼可见**。

## O1. 教学骨架生成 ⭐

**🎯 Debate**
- Proposer：让 LLM 全自动从 atom 聚类生成骨架。
- Reviewer：自下而上聚类已经被证明无效（翻车样本）。骨架必须**自上而下**——基于学科常识 + 第一篇素材主题。且学生应有**确认权**。
- Host 决议：第一次进入 library 时强制走骨架引导（约 1–2 分钟），是整个系统**唯一一次较重的介入**。

**目标**：为每个 library 建立一份 5–10 条 mental_model 的"教学骨架"，作为后续 atom 归位的坐标系。

**输入**：
- library 名称（如"宏观经济学"）
- library 内已有 asset（至少 1 篇）
- 可选：学生提供的教学大纲 PDF / 课程描述

**输出**：
- `Skeleton { library_id, models: [{id, title, core_question, status}] }`
- 5–10 条骨架点，每条包含：
  - title（如"货币政策传导机制"）
  - core_question（如"央行的政策行为如何影响实体经济？"）
  - status ∈ `{pending, has_atoms, well_supported}`

**使用场景**：
1. 第一次为"宏观经济学"创建库：导入 PPT → 系统建议骨架 → 学生确认
2. 学生中途换教材，骨架需重生成

**核心交互逻辑**：
1. 触发：library 首次有 ≥1 个 asset 完成 E2 抽取
2. LLM 调用：基于学科常识 + 已抽 atom + 可选大纲 → 输出 5–10 条 model 候选
3. UI:骨架确认页面
   - 每条 model 一张卡片，可一键删除
   - 底部"+ 新增 model"按钮
   - 学生可拖拽排序
   - 顶部 [使用建议骨架] / [我自己列] / [跳过，先用扁平结构]
4. 确认 → 写入 skeleton 表，触发 O2 反向挂载

**介入级别**：🤝（系统出方案 + 学生 1–2 分钟编辑）

**边界约束**：
- 5 条以下报"骨架太薄，建议补素材后再生成"
- 10 条以上自动合并最相似的两条
- 学生选[跳过]→ 库进入"扁平模式"，所有 atom 直接列表展示，不强求骨架；学生任何时刻可触发"现在生成骨架"

**优先级**：P0（**最大的根因修复**）

## O2. Atom 反向挂载

**🎯 Debate**
- Proposer：每个 atom 自动挂到最相似 model 即可。
- Reviewer：相似度低于阈值时不能强行挂——需要进 unsorted 池子让学生决定。
- Host 决议：阈值制 + 兜底池。

**目标**：让每个新抽出的 atom 找到骨架上的归属，没归属的进 unsorted 池。

**输入**：
- E2 输出的 atom
- 当前 library 的 skeleton

**输出**：
- atom.candidate_skeleton_id（高置信时直接挂载）
- 或 `unsorted` 状态（待 O4 处理）

**使用场景**：每次 E2 抽出新 atom 都自动触发。

**核心交互逻辑**：
1. atom embedding ↔ 每个 model 的 core_question + 已挂 atom 的平均向量
2. 最高相似度 > 0.7 → 直接挂载 + UI 上 model 计数 +1
3. 0.4–0.7 → 挂载但标 "tentative"，O4 可推送给学生
4. < 0.4 → 进 unsorted 池

**介入级别**：🤖

**优先级**：P0

## O3. KU 合成

**🎯 Debate**
- Proposer：KU = 聚类后命名（当前实现）。
- Reviewer：KU 应该是"骨架点的自然形态"——一条 model + 它支撑的 atom 集合，不需要额外聚类。
- Host 决议：KU = skeleton.model + 其挂载的 atom，**自动派生，不再聚类**。

**目标**：把骨架点 + 挂载 atom 包装成可学习的 KU 对象。

**输入**：
- skeleton.model
- 该 model 下挂载的 atom 列表
- 关联 source_excerpts

**输出**：
```ts
KnowledgeUnit {
  id, library_id
  skeleton_model_id  // 直接对应
  title              // 来自 skeleton.model.title
  core_insight       // LLM 综合 atom 的 why_it_matters 生成
  summary            // E7-equivalent 复用
  status             // 见 v1.0 7 级状态机
  constituent_atom_ids[]
  source_asset_ids[]
}
```

**使用场景**：每次 model 下首次挂上 atom，自动创建 KU；后续 atom 累加更新 KU。

**核心交互逻辑**：
1. O2 挂载完成后 → 检查该 model 是否已有对应 KU
2. 无 → 创建 KU；有 → 更新 KU 的 atom 列表 + 重新生成 core_insight 与 summary
3. KU 永远是 model 的"实体化"，不允许独立于 model 存在

**介入级别**：🤖

**优先级**：P0

## O4. 未挂载 atom 推送

**🎯 Debate**
- Proposer：unsorted 池子让学生进库自己看。
- Reviewer：太被动。学生不知道 unsorted 存在。需要在合适时机推送。
- Host 决议：积累到 ≥ 3 个时在 Home 通知；学生主动开任务时也提示。

**目标**：让学生定期处理无法自动归位的 atom——决定它是新 model / 噪声 / 边缘案例。

**输入**：unsorted 池中累积 ≥ 3 个 atom，且距上次推送 ≥ 7 天。

**输出**：用户的决定动作之一：
- 创建新 model（O5 修订）
- 强制挂到现有 model（覆盖低置信判断）
- 标"暂存"（不再推送）
- 删除 atom

**使用场景**：每周一次（不强制），Home 出现「有 5 个 atom 还没归位，看一下？」

**核心交互逻辑**：
1. Home 通知 → 点击进 unsorted 处理页
2. 一次显示一条 atom，4 个动作按钮
3. 完成后下一条；ESC 退出
4. 全部处理完 → "处理完成"

**介入级别**：🤝

**边界约束**：
- 最多积累 20 条不推；超 20 立刻在 Home 提示"unsorted 太多，建议生成新 model"
- 用户跳过整个推送 → 30 天后再来

**优先级**：P1

---

## O5. 骨架修订

**🎯 Debate**
- Proposer：骨架一旦定型不要动。
- Reviewer：学一段时间后学生会发现骨架缺了/多了，必须可改。
- Host 决议：允许修订，但有低摩擦的"影响预览"——告诉学生"修改后会影响 X 个 KU"。

**目标**：学生可在任何时候增/删/合并/拆分骨架的 model。

**输入**：用户主动操作（知识模式 → 骨架视图 → 编辑）。

**输出**：
- 新增 model：触发 O2 重新跑 unsorted 池
- 删除 model：该 model 的 KU 标 archived；其挂载 atom 退回 unsorted
- 合并：两 model 的 atom 合一，KU 保留主标题，副 KU archived
- 拆分：LLM 协助提议拆分边界，学生确认

**使用场景**：
1. 学到一半发现"国际金融"和"国际贸易"应该分两条 model
2. 期末复习时发现某条 model 从没挂上 atom，删除

**核心交互逻辑**：
1. 骨架视图 → 长按 model 卡片 → 弹四个动作 [改名] [合并] [拆分] [删除]
2. 删除/合并 → 显示影响预览「将影响 3 个 KU 和 12 个 atom，确认？」
3. 确认后异步跑后处理

**介入级别**：✋

**优先级**：P1

---

## O6. KU 命名校准

**🎯 Debate**
- Proposer：LLM 生成的 title + core_insight 不够时学生可手改。
- Reviewer：和 O5 重复？
- Host 决议：不重复——O5 是改 model 骨架点；O6 是改 KU 实体的展示文案。两层独立。

**目标**：学生可改单个 KU 的 title / core_insight，覆盖 LLM 默认。

**输入**：用户在 KU 详情页点编辑。

**输出**：
- KU.title 与 core_insight 写入用户覆盖值
- 标 `user_edited = true`，后续不被 LLM 重写

**使用场景**：学生觉得 LLM 生成的 title 不够准/不够顺口。

**核心交互逻辑**：
1. KU 详情页右上角铅笔图标 → 进入编辑模式
2. 直接编辑 title / core_insight
3. 保存 → 写入 user_edited 标记

**介入级别**：✋（低频）

**优先级**：P2

# 第三部分：Record 阶段（知识记录）

> **阶段价值**：把"看懂"变成"我懂"。学生用自己的话沉淀一遍 + 接收 AI 镜子反馈，把外部知识转成自己能调用的版本。
>
> **学生在这阶段拿到的回报**：自己的"私版"理解被沉淀下来，将来回看不是抄书摘要，是**当年自己的话**。

---

## R1. 用自己的话解释 ⭐

**🎯 Debate**
- Proposer：可写文字也可录音，多种形式更友好。
- Reviewer：形式无所谓，**关键是不能让 AI 代写**——一旦 AI 写完用户改两句，"沉淀"就名存实亡。
- Host 决议：必须用户原创；可文字 + 语音；禁止 AI 提供"建议初稿"。

**目标**：让学生用自己的话重述每个 KU，作为内化的核心动作。

**输入**：
- KU 上下文（title + core_insight + 学生已读的 summary）
- 用户输入（文字或语音转录）

**输出**：
- `UserExplanation { ku_id, text, source: 'text'|'voice', length, timestamp }`
- 触发 R2 镜子反馈

**使用场景**：
1. 看完 KU 摘要后，下方编辑区写自己的理解
2. 通勤时用语音记录对某个 KU 的当前理解

**核心交互逻辑**：
1. KU 详情页"第三步：用你自己的话说"区域，默认空白
2. 输入框 + 切换语音按钮
3. 至少 50 字（或 30 秒语音）才允许保存
4. 保存 → 自动触发 R2 镜子，且不可撤销保存（防止用户依赖"先看 AI 反馈再写"）

**介入级别**：✋（**核心动作，不可代替**）

**边界约束**：
- **不提供 AI 起草**——这是产品哲学红线
- 不显示"模板/示例"诱导抄袭
- 字数下限是软约束，可调

**优先级**：P0

---

## R2. AI 镜子反馈

**🎯 Debate**
- Proposer：当前 mirror_feedback 设计已成熟，沿用。
- Reviewer：sole change：把对比对象从 concept 切到 KU 的 atom 集合，prompt 文本不动。
- Host 决议：直接复用 v1.0 prompt #12，仅切输入。

**目标**：温和对比学生表述与文档关键点，呈现"你说到了 X / 文档里还有 Y"。

**输入**：
- R1 的 user_explanation
- KU 关联的 atom 列表（statement + why_it_matters + excerpts）

**输出**：
```ts
MirrorFeedback {
  covered_count: number
  covered_points: string[]
  additional_perspectives: { text, source }[]
  difference_note: string | null
}
```

**使用场景**：R1 保存后立刻触发，3–5 秒返回。

**核心交互逻辑**：
1. R1 保存 → 异步调 LLM
2. 返回后在 R1 编辑区下方展开反馈卡片
3. covered_points 高亮（绿色）；additional_perspectives 灰色卡片
4. 学生可点 additional 进 R3 补充

**介入级别**：🤖

**边界约束**：
- 永不出现"对/错/不完整/缺少"等评判性词汇（v1.0 强约束）
- 反馈失败 → 显示"今天镜子有点累，稍后再试"，不阻断 R1 保存

**优先级**：P0

---

## R3. 开放式补充笔记

**🎯 Debate**
- Proposer：和 R1 合并即可。
- Reviewer：R1 是"重述本 KU"，R3 是"我对这个 KU 的所有想法"——结构化与开放式应分开。
- Host 决议：分两区——R1 重述（核心动作）+ R3 自由笔记（开放）。

**目标**：让学生记录题外话、跨课程联想、个人例子等无结构内容。

**输入**：用户在 KU 详情页"自由笔记"区输入。

**输出**：
- `UserNote { ku_id, text, attached_assets[], created_at, updated_at }`

**使用场景**：
1. 学生想到一个个人理财例子，想记下来挂在某 KU 上
2. 学生写下"这个理论我觉得有问题，因为..."

**核心交互逻辑**：
1. KU 详情页可折叠"我的笔记"区域
2. 富文本编辑器（最简的 Markdown 即可）
3. 实时保存（防抖 2 秒）
4. 可拖拽 asset 进笔记作为引用

**介入级别**：✋

**优先级**：P1

---

## R4. 理解快照保存

**🎯 Debate**
- Proposer：每次 R1 保存都创建快照。
- Reviewer：会产生太多快照噪声。应该有节流。
- Host 决议：节流——R1 保存触发 + 距上一快照 ≥ 24h 才创建。

**目标**：在用户每次"重要更新"理解时存一份当时的版本，作为 V12 时间线原料。

**输入**：
- R1 的 user_explanation 内容变化
- R2 的 mirror feedback 结果

**输出**：
```ts
UnderstandingSnapshot {
  id, ku_id, timestamp
  user_explanation, mirror_covered_count, mirror_missed_areas[]
  depth_level, source_atom_count_at_time
}
```

**使用场景**：背景行为，用户不主动感知。

**核心交互逻辑**：
1. R1 保存触发器
2. 检查最近一次快照时间戳：< 24h 内 → 跳过；≥ 24h → 创建
3. 存入数据库，无 UI 反馈

**介入级别**：🤖

**优先级**：P1

---

## R5. 深度评级

**🎯 Debate**
- Proposer：完全自动评级。
- Reviewer：LLM 评级误差大，学生自评更准。但全靠学生评又增负担。
- Host 决议：LLM 出建议 + 学生一键确认/调整。

**目标**：每次理解快照后给当次理解打 ★1–5，让学生看到深度变化。

**输入**：
- R1 user_explanation
- R2 mirror feedback（covered_count、difference_note）
- 历史快照对比

**输出**：
- `depth_level: 1–5`
- 显示在 KU 详情页 + 时间线

**使用场景**：每次 R1+R2 完成后弹出。

**核心交互逻辑**：
1. R2 反馈展示后，下方出现「这次理解的深度：建议 ★★★☆☆」+ [接受] [改为] 按钮
2. 改为 → 1–5 滑块
3. 不操作 → 5 秒后自动接受建议值

**介入级别**：🤖 + 🤝

**优先级**：P1

# 第四部分：Evolve 阶段（知识进化）

> **阶段价值**：进化由**真实使用场景**触发，不由日历强制。四种触发模式：A 考试 / B 任务 / C 阅读 / D 回望。
>
> **修订声明**：本部分对 v1.0 宪章 §7「进化引擎」整章构成正式修订。删除"每天 9 点 cron 自动推送"、"本地通知催复习"等行为；SM-2 间隔算法仅在考试模式下启动。

---

## V1. 任务会话开启 ⭐

**🎯 Debate**
- Proposer：任务会话需要复杂的 setup（选 KU、选输出格式）。
- Reviewer：setup 越复杂，学生越不开。任务描述应该 ≤ 1 句话。
- Host 决议：极简——一行任务描述 + 自动检索。

**目标**：学生用 10 秒描述任务，进入任务全屏模态。

**输入**：
- 任务描述文本（1–3 句，必填）
- 任务类型（可选）：写作 / 分析 / 复习 / 解题 / 其他
- 关联 library（可选，默认全库）

**输出**：
- `TaskSession { id, description, type, library_ids[], started_at, status: 'active' }`

**使用场景**：
1. 写论文前打开任务："写一篇关于美联储 2024 加息周期对新兴市场影响的分析（2000 字）"
2. 做作业前打开任务："分析这道题：货币政策传导失效的微观机制"

**核心交互逻辑**：
1. Home 点 [💬 开始任务] → 弹一个**单输入框 + 类型 chip**
2. 输入完 → 回车 → 立刻进入全屏任务模式（不要二级确认）
3. 后台同步触发 V2 检索

**介入级别**：✋（10 秒）

**优先级**：P0

---

## V2. 任务相关 KU 检索

**🎯 Debate**
- Proposer：检索全库返回 top 20。
- Reviewer：20 个太多，学生看不过来。要分层——"高相关 / 边缘相关 / 缺口"三层。
- Host 决议：top 5 高相关 + 3 边缘 + 缺口提示。

**目标**：用任务描述当 query，从用户 KU 库检索语义相关条目。

**输入**：
- TaskSession.description
- 用户的全部 KU embedding

**输出**：
- 高相关：cosine > 0.65 的 KU，top 5
- 边缘相关：0.45–0.65，top 3
- 缺口：基于任务描述但无对应 KU 的概念，top 2

**使用场景**：V1 创建后立刻触发，3 秒内填充任务模态右侧栏。

**核心交互逻辑**：
1. 任务模态右侧"工具箱"区先显示 skeleton 加载态
2. 检索完成 → 三段式列表（高/边缘/缺口）
3. 每个 KU 卡片含 title + core_insight + [展开] [拖入草稿] 按钮

**介入级别**：🤖

**优先级**：P0

---

## V3. 知识缺口提示

**🎯 Debate**
- Proposer：缺口列得越详细越好。
- Reviewer：太多缺口 = 自我质疑，反作用。最多 2 条。
- Host 决议：2 条上限。

**目标**：让学生看到"任务里有什么是我现在没有的"，决定是否补。

**输入**：
- 任务描述
- V2 已找到的 KU
- LLM 提示："该任务可能还需要哪些这个学生还没建立的概念/model？"

**输出**：≤ 2 条缺口提示，每条含：
- gap_concept_name
- why_relevant
- 建议动作 [跳过] [搜素材] [先记下来]

**使用场景**：V2 完成后立刻附在工具箱底部。

**核心交互逻辑**：
1. 缺口卡用警示色（黄色，非红色）
2. [搜素材]→ 弹素材搜索框（库内 + 可选外部资源）
3. [先记下来]→ 在任务模态记一笔，结束时入 unsorted 池

**介入级别**：🤝

**优先级**：P1

---

## V4. 任务中 KU 套用与组合 ⭐

**🎯 Debate**
- Proposer：直接拖 KU 进草稿即可。
- Reviewer：拖完之后？学生需要写"这条 KU 在我这次任务里怎么用"——这才是融合。
- Host 决议：拖入触发"写一句这条 KU 怎么帮你的"小窗，可跳过。

**目标**：让学生显式写下"这条旧 KU 在新任务里如何被套用"——这是认知融合发生地。

**输入**：
- 学生从工具箱拖 KU 进草稿
- 学生在草稿中引用 KU

**输出**：
```ts
KUUsage {
  task_session_id, ku_id
  applied_note: string   // 学生写的"如何套用"
  position_in_draft: number
  created_at
}
```

**使用场景**：
1. 学生拖"泰勒规则"进论文草稿，写"用泰勒规则检验美联储 12 月利率决定的反应函数"
2. 学生在草稿里 @KU 引用，弹出"如何套用"输入

**核心交互逻辑**：
1. 拖入草稿 → 浮窗 "用一句话说说它怎么帮你？" + 输入框 + [跳过]
2. 写完保存 → 草稿里出现 KU 的 inline reference + tooltip 是 applied_note
3. 跳过 → KU 仅作为 reference，不生成 KUUsage 记录

**介入级别**：🤝（可跳过但鼓励填）

**优先级**：P0

---

## V5. 反例标记

**🎯 Debate**
- Proposer：让学生主动标"这条 model 在这里不灵"。
- Reviewer：学生可能不知道"反例"这个术语。系统应主动识别。
- Host 决议：双轨——LLM 检测候选 + 学生确认。

**目标**：捕获"旧 model 在新场景失效"的案例，作为修订线索。

**输入**：
- KU 的 statement
- 学生在草稿里关于这条 KU 的语句
- 简单的反义词/转折语义检测（如"但是""不适用""失效""反例"）

**输出**：
- `Counterexample { ku_id, task_session_id, excerpt, user_confirmed: bool }`
- KU 上挂"⚠ N 个反例"badge

**使用场景**：学生在草稿写"泰勒规则在零利率下界时不适用"——系统标黄并问"是反例吗？"

**核心交互逻辑**：
1. 草稿监听关键词触发器
2. 命中 → 在该段落右侧出现小黄旗"这是 [泰勒规则] 的反例吗？" + [是] [否，只是讨论]
3. [是] → 写入 Counterexample 表，KU 详情页"反例"区可见

**介入级别**：🤝

**优先级**：P2

## V6. 任务结束留痕

**🎯 Debate**
- Proposer：关闭任务时弹大表单确认所有留痕项。
- Reviewer：关任务的瞬间学生最不耐烦确认。应该全自动 + 关后小总结。
- Host 决议：自动留痕，关后展示 30 秒小总结，可点撤销。

**目标**：任务关闭后自动写入所有衍生数据。

**输入**：TaskSession 的所有过程数据。

**输出**：
- 每个被使用的 KU 增加 ActivationSnapshot
- 草稿中产生的新概念入 unsorted 池（候选 atom）
- KUUsage 全部入库
- 反例确认入库
- TaskSession.status = 'completed'

**使用场景**：学生点[完成]或关闭任务模态。

**核心交互逻辑**：
1. 点[完成]→ 后台异步留痕
2. 跳到 Home，顶部出现 30 秒卡片：「任务 [X] 完成。激活 5 个 KU，新发现 2 个候选概念」+ [查看] [撤销]
3. 30 秒后自动消失

**介入级别**：🤖

**优先级**：P0

## V7. 阅读触发反查 ⭐

**🎯 Debate**
- Proposer：每次导入新 asset 都反查。
- Reviewer：和 E2 抽取冲突——两者都在导入触发，会不会重？
- Host 决议：两者都跑，但分两条。E2 输出新 atom，V7 输出"这篇文章用到了你已有的 X、Y、Z"。两条不互相替代。

**目标**：导入新文章时反过来对照已有 KU，告诉学生"这篇文章踩了你哪几条 model"。

**输入**：
- 新导入 asset 的 rawText
- 用户已有的全部 KU

**输出**：
```ts
ReadingActivation {
  asset_id
  matched_kus: { ku_id, snippet, match_reason }[]
  new_concepts: string[]      // 文章里有但用户库没有的
  contradictions: { ku_id, snippet }[]  // 文章和用户某 KU 冲突
}
```

**使用场景**：学生导入一篇 FT 评论文章，系统告诉他"这篇用到了你的'泰勒规则'和'不可能三角'，且作者反对你笔记里的'通胀目标制'立场"。

**核心交互逻辑**：
1. E2 完成后并行触发 V7
2. 完成 → Home 静默通知「这篇文章激活了你的 3 个 KU，有 1 处分歧」
3. 点击 → 进入"阅读反查"页面
   - 文章原文左 + 激活高亮
   - 右侧 KU 卡片堆叠
   - 分歧用红色标注 + [展开成任务] 按钮
4. [展开成任务] → 自动创建 TaskSession 进任务模式

**介入级别**：🤖（推送）+ 🤝（学生可选展开）

**优先级**：P0（**最高 ROI 新功能**）

## V8. 考试目标创建

**🎯 Debate**
- Proposer：和 Calendar 集成，从课表自动识别考试。
- Reviewer：自动识别考试日期容易出错。手动创建更可靠。
- Host 决议：手动为主，Calendar 自动检测仅作为建议。

**目标**：学生定义一个考试目标，包括日期、范围、难度。

**输入**：用户在考试模式点[创建考试]。

**输出**：
```ts
ExamTarget {
  id, name, exam_date
  ku_ids[]              // 范围
  difficulty: 'recall' | 'understand' | 'apply'
  status: 'active' | 'completed' | 'archived'
  created_at
}
```

**使用场景**：
1. 期中考前 3 周创建"宏观经济学期中"
2. 考研冲刺创建多个考试

**核心交互逻辑**：
1. 表单：名字 + 日期（日历选择器）+ 范围（多选库 / 多选 KU）+ 难度
2. 创建后 → 自动算调度（V9）+ 跳转考试详情页

**介入级别**：✋

**优先级**：P1

## V9. 考试反向调度推送

**🎯 Debate**
- Proposer：每天推送 N 张卡。
- Reviewer：考试期前 30 天和前 3 天的强度应该不同。
- Host 决议：分阶段——60+/30/14/3 天四档强度。

**目标**：在考试期内分阶段推送复习卡片。

**输入**：ExamTarget + 当前日期。

**输出**：
- 当日推送队列（卡片列表）
- 推送形式：考试模式首页 + 可选系统通知

**使用场景**：考试期内每天打开 App 看到当日队列。

**核心交互逻辑**：
1. 距考试 60+ 天：每周 2 次，每次 5 张
2. 30 天内：每天 1 次，每次 5 张
3. 14 天内：每天 1 次，每次 8 张 + 模拟题模式
4. 3 天内：冲刺模式，全 KU 滚动复习
5. 学生选[做]→ 卡片正反面，答完一键评分（自评：会 / 模糊 / 不会）
6. 自评结果回写 SM-2 间隔（仅本考试目标内有效）
7. 考试日过后 → 推送停止，进入"已结束"档案

**介入级别**：🤖（推）+ ✋（学生选做）

**优先级**：P1

## V10. 自主回望

**🎯 Debate**
- Proposer：随机选 KU 展示。
- Reviewer：随机会有"刚学的"被反复推。应"很久没用 + 当时深度高"加权。
- Host 决议：加权随机。

**目标**：在无目的时刻给学生一份"老相册"般的回放。

**输入**：用户主动点[🕐 时光]→ [随机回放]。

**输出**：
- 单个 KU 的展示卡：当年快照 + 当时笔记 + R5 深度 + 已多久没碰

**使用场景**：周末刷会儿，看看以前学的东西。

**核心交互逻辑**：
1. 时光视图入口提供[随机回放]按钮
2. 点击 → 全屏卡片展示一条 KU
3. 底部 [换一个] [我现在看看] [关闭]
4. [我现在看看]→ 跳转 KU 详情页

**介入级别**：✋（用户主动）

**优先级**：P2

## V11. Staleness 检测

**🎯 Debate**
- Proposer：新素材进库后立刻提示对应 KU 更新。
- Reviewer：太频繁会让学生疲劳。应该批量+延迟提示。
- Host 决议：积累 3 个新 atom 或 7 天后批量提示。

**目标**：当 KU 关联的素材有新增时，温和提示学生是否更新。

**输入**：
- KU 的 source_asset_count
- 实际关联 atom 数

**输出**：KU 详情页顶部 toast 「📄 有 N 个新素材未纳入摘要」+ [更新] [暂时不动]

**使用场景**：学生重新打开某 KU，看到提示。

**核心交互逻辑**：
1. 后台定期扫 staleness
2. KU 详情页打开时检查 → 顶部出现非阻塞 toast
3. [更新]→ 后台重生成 summary，复用 prompt #10
4. [暂时不动]→ 30 天内不再提示

**介入级别**：🤖（检测）+ 🤝（学生决定）

**优先级**：P1

## V12. 理解时间线展示

**🎯 Debate**
- Proposer：嵌入 KU 详情页。
- Reviewer：也应有库级时间线（看整库的进化）。
- Host 决议：双层——KU 内嵌 + 时光视图全库。

**目标**：让学生看到自己的话怎么变好的。

**输入**：UnderstandingSnapshot 的时间序列。

**输出**：
- KU 详情页内的时间轴：每个快照点显示日期 + R5 深度 + 摘录前 30 字
- 时光视图全库：聚合所有 KU 的快照，按时间排开

**使用场景**：复习时看自己 6 个月前怎么想的。

**核心交互逻辑**：
1. KU 详情页折叠的"理解历史"区
2. 展开 → 时间轴 + 卡片，可点击展开当时全文
3. 时光视图 = 全库版本

**介入级别**：🤖（展示）

**优先级**：P2

## 5.2 关键交互模式总结

| 学生动作 | 交互形态 | 时间预算 |
|---|---|---|
| 导入素材 | 拖拽 | 5 秒 |
| Big Idea 标注 | 一行输入或 30s 语音 | 10 秒（可跳过）|
| 库归属确认 | 二选一 toast | 2 秒（仅低置信时）|
| 骨架确认 | 编辑卡片 | 1–2 分钟（一次性）|
| 用自己的话 | 文字/语音 | 3–10 分钟 |
| 任务开启 | 一行描述 | 10 秒 |
| 拖 KU + 写套用 | 拖 + 一句话 | 30 秒 |
| 任务关闭 | 点完成 | 1 秒 |
| 考试创建 | 表单 | 1 分钟 |
| 考试卡复习 | 翻卡 + 自评 | 5 分钟/天 |

学生 95% 的时间都花在 R1（用自己的话）和 V4（任务中套用）上——这正是产品价值的根。其它环节加起来日均 < 2 分钟。

# 附：术语表

| 术语 | 定义 |
|---|---|
| **Atom** | 强类型最小知识单位。5 类：mental_model / method / claim / exemplar / distinction |
| **Concept** | 名词性词条，被 atom 引用作为内部零件，不再作为用户可见单元 |
| **Skeleton** | library 级别的"教学骨架"，由 5–10 条 mental_model 构成，是 atom 的归位坐标系 |
| **Knowledge Unit (KU)** | 骨架点的实体化形态——一条 model + 它支撑的所有 atom |
| **Big Idea Signal** | 学生在课后用 10 秒提供的"这次最重要"的强信号 |
| **Task Session** | 任务驱动模式（Mode B）下的全屏工作会话，是认知融合发生地 |
| **KU Usage** | 学生在任务中显式写下的"这条 KU 在新场景如何被套用"的记录 |
| **Activation Snapshot** | 任务结束时为每个被使用的 KU 留下的"被调用了一次"的痕迹 |
| **Reading Activation** | 导入新素材时反查已有 KU 的输出（"这篇文章用到了你的 X、Y、Z"）|
| **Exam Target** | 学生定义的考试目标，是 SM-2 间隔复习算法**唯一**的启动入口 |
| **Unsorted Pool** | 抽取的 atom 中无法自动归骨架的兜底池，由 O4 推送学生处理 |
