# Debate Log — 日历功能迭代

> **辩题**：如何迭代 NCdesktop 日历功能，使用户在「当天无课」场景下仍能看到日历入口，并通过日历视图浏览和预习未来的每一堂课？
>
> **Session**: notecapt 迭代 / debate / session_001
> **启动时间**: 2026-04-11
> **参与角色**: Host · Proposer · Reviewer

---

## 初始项目状态

### 当前日历功能实现概述

NCdesktop 的日历功能（v2.1）目前包括以下模块：

**前端层：**
- `CourseSection.tsx`（侧边栏课程分区）：按 Today / Tomorrow / This Week 三组展示课程。**当三组均为空时 `return null`，整个分区不渲染。**
- `CourseEventItem.tsx`：单条课程行（时间 + courseCode/title）
- `CalendarImportTab.tsx`：设置面板中的「课程日历」标签页，支持拖入 .ics 文件 / URL 订阅 / 预览勾选 / 确认导入
- `CoursePreviewSpace.tsx`：AI 课程预习空间（ContentArea 的一个视图模式），含 AI 预习指南 + 用户笔记

**状态管理层：**
- `calendarStore.ts`：持有 `events: CourseEvent[]`、导入预览状态、派生方法（getTodayEvents / getTomorrowEvents / getThisWeekEvents / getEventsGroupedByDay）
- `uiStore.ts`：`activeCourseEventId` + `rightPanelMode`（含 `"course_preview"` 枚举值）

**后端层（Rust / Tauri）：**
- `commands/calendar.rs`：6 个 IPC 命令（import_ics_file / import_ics_url / confirm_import_events / get_course_events / delete_calendar_source / refresh_ics_subscription）
- `commands/course_preview.rs`：3 个命令（generate_course_preview / get_course_preview / save_preview_notes）
- `db/calendar.rs`：CourseEvent CRUD + 按 source_uid 去重
- `db/course_preview.rs`：CoursePreview upsert / 查询
- `ics_parser.rs`：ICS 文件解析

**数据库表：**
- `course_events`：课程事件（id, library_id, project_id, title, course_code, instructor, location, start_time, end_time, recurrence_rule, day_of_week, ...）
- `course_previews`：AI 预习内容（id, course_event_id, content, user_notes, model, generated_at, ...），与 course_events 一对一

### 当前用户交互流程

```
用户在「设置 → 课程日历」中导入 .ics 文件或 URL
    ↓
侧边栏 CourseSection 显示 Today / Tomorrow / This Week 分组课程
    ↓
用户点击某课程 → ContentArea 切换为 CoursePreviewSpace
    ↓
CoursePreviewSpace 自动生成/加载 AI 预习指南 + 用户可编辑笔记
    ↓
用户点击 Back → 返回常规视图
```

### 核心问题

1. **入口消失**：`CourseSection` 第 49-55 行，当三组事件全部为空时 `return null`，日历入口从侧边栏彻底消失。用户在假期、周末、无课日等场景下完全无法访问日历功能。

2. **缺少日历视图**：没有任何日历 UI（周视图 / 月视图 / 议程视图），用户无法浏览未来课程的全景。

3. **层级扁平**：点击课程直接跳入 AI 预习空间，没有中间的「课程概览」层级。用户无法快速浏览课程信息而不触发 AI 生成。

4. **数据加载范围受限**：`fetchEvents` 默认加载 `now` 到 `now+7天` 的数据，无法看到更远期的课程。

## Layer 1 — 问题定义

### Round 1 - Layer 1

**Host 开场问题给 Proposer：**

请回答以下核心问题：
1. 日历功能迭代要解决的核心问题究竟是什么？请区分「入口可见性」和「日历浏览能力」两个维度。
2. 目标用户是谁？他们在什么场景下需要日历？
3. 系统边界：这次迭代涵盖哪些内容，哪些明确不做？
4. 成功的定义是什么？给出可度量的标准。

---

**Proposer：**

### 1. 核心问题定义

这次迭代要解决**两个紧密相关但层次不同的问题**：

**问题 A — 入口可见性（Availability）**
> 用户在「当天无课」场景下（周末、假期、休息日），侧边栏日历入口完全消失，导致用户无法主动进入日历相关功能。这违反了「功能入口应当始终可达」的基本可用性原则 `[最佳实践]`。

**问题 B — 日历浏览能力（Browsability）**  
> 用户缺乏一个日历视图来浏览未来课程的全景。当前只有扁平的 Today/Tomorrow/ThisWeek 列表，无法回答「下周三有什么课？」「这学期的课程分布是怎样的？」等问题。预习材料的入口过于直接——点击课程就触发 AI 生成，用户无法「先看看有什么课」再决定要不要预习。

**两者的关系**：A 是 B 的前提（入口都看不到，更谈不上浏览），但 A 单独解决（只是始终显示入口）并不充分——如果入口进去后还是只能看到「今天没有课」的空状态，用户体验提升有限。

### 2. 目标用户与场景

**主要用户**：大学生 / 研究生，使用 NoteCapt 管理课程笔记和知识资产。

**核心场景**：

| 场景 | 描述 | 频率 |
|------|------|------|
| S1 周末预习 | 周末想提前预习下周课程，查看日历找到目标课程 | 每周 1-2 次 |
| S2 无课日浏览 | 当天没课，但想看看本周还有什么课需要准备 | 每天可能 |
| S3 学期规划 | 学期初导入课表后，浏览整个学期的课程分布 | 学期初 1-2 次 |
| S4 当日预习 | 课前 30 分钟，快速查看今天的课并进入预习 | 每个上课日 |

**S4 是当前已经较好支持的场景**（前提是当天有课）。**S1、S2、S3 是本次迭代要重点解决的**。

### 3. 系统边界

**做什么（In Scope）**：
- 侧边栏日历入口始终可见，即使没有任何课程事件
- 新增日历视图（周视图为核心），在 ContentArea 中渲染
- 建立「日历入口 → 日历视图 → 课程预习」的三层导航
- 扩展数据加载范围，支持加载任意时间段的课程

**不做什么（Out of Scope）**：
- 不做日历事件编辑/创建（课程来源仍为 ICS 导入）
- 不做日历提醒/通知（留待后续迭代）
- 不做月视图/年视图（MVP 只做周视图）
- 不做与系统日历（Apple Calendar）的双向同步
- 不做移动端适配

### 4. 成功标准

| 标准 | 度量方式 |
|------|---------|
| 日历入口始终可见 | 在无课程事件时，侧边栏仍然显示日历入口 |
| 日历视图可浏览 | 用户可以通过周视图查看任意一周的课程分布 |
| 预习入口可达 | 从日历视图中点击任意课程可进入预习空间 |
| 导航层级清晰 | 用户可以在「日历视图 ↔ 课程预习」之间自由切换 |
| 渲染性能 | 周视图加载 < 200ms，切换周 < 100ms |

**脆弱性声明**：
> 这个方案在以下条件下可能失效：
> - 用户有极大量课程（> 500 事件/学期），周视图渲染可能卡顿
> - 用户跨时区使用，日期分组逻辑可能出错

**Reviewer：**

**挑战层级：L2（结构）+ L3（根本）**

我对 Proposer 的问题定义有以下挑战：

**挑战 1（L3 根本）：「日历」的定义是否准确？**

> 你把这个功能称为「日历」，但实际上用户导入的是**课表**（academic schedule），它和通用日历有本质区别：课表是**周期性重复**的结构化数据，而通用日历是**一次性事件**的集合。你设计的「周视图」到底是在展示一个日历，还是一个课表？这个定义会直接影响 UI 的信息架构——课表周视图应该强调**时间槽和重复模式**，而日历周视图应该强调**具体日期和一次性事件**。

**挑战 2（L2 结构）：S3「学期规划」场景是否真的在 scope 内？**

> 你把 S3 列为核心场景，但又在 Out of Scope 中说「不做月视图/年视图」。一个只有周视图的日历如何支撑「浏览整个学期的课程分布」这个场景？如果 S3 无法被 MVP 覆盖，是否应该从核心场景中移出，改为 P2？

**挑战 3（L2 结构）：「三层导航」是否引入了不必要的复杂度？**

> 你提出「日历入口 → 日历视图 → 课程预习」的三层导航。但用户在 S4（当日预习）场景下，已经习惯了「侧边栏点击课程 → 直接进入预习」的两步流程。三层导航会让 S4 变成三步，增加了日常高频操作的摩擦。你如何平衡「新增日历浏览」和「不破坏现有快速预习流程」之间的矛盾？

**隐含假设探测**：
> 你的方案假设用户的课表是通过 ICS 导入的、结构化的数据。但如果用户只导入了部分课程（例如只导了主课没导选修），日历视图会呈现一个「半空」的状态，这种体验是否反而比当前的列表更差？

**Proposer：**

> **Reviewer 挑战 1**：「日历 vs 课表」的定义
>
> 你说得对，这个定义至关重要。我将方案中的用语从「日历视图」调整为**「课程日历视图」（Course Calendar View）**。

**定义**：课程日历视图是一个**以学术周为单位的课表浏览器**，它展示的是**周期性课程事件在时间槽上的分布**，而非通用日历事件。

**对 UI 的影响**：
- 周视图的列标题应该是**星期几**（Mon-Sun），而非具体日期
- 可以支持「典型周」视图（展示课程的周期模式）和「具体周」视图（展示某一周的实际课程），默认为「具体周」

- 当有课程变动（取消/调换）时，应该在卡片上标注差异

这个定义的边界是：**只展示从 ICS 导入的课程事件**，不处理个人日程、日记、TODO 等非课程数据。`[项目约束]`

> **Reviewer 挑战 2**：S3 场景在 MVP 中的定位
>
> 关于 S3「学期规划」，你说得对——只有周视图确实无法完整支撑「浏览整个学期」的场景。

**方案调整**：将 S3 从核心场景降级为 P2。MVP 的核心场景为：
- **S1 周末预习**（P0）：周视图 + 课程预习入口
- **S2 无课日浏览**（P0）：始终可见的入口 + 周视图
- **S4 当日预习**（P0）：保持现有快速路径

S3 可以在后续通过月视图或学期概览来支撑。

> **Reviewer 挑战 3**：三层导航 vs 现有快速流程
>
> 这是一个有效的挑战。我的立场是：**两条路径并存，而非替代**。

**具体方案**：

| 路径 | 流程 | 覆盖场景 |
|------|------|---------|
| **快速路径（保留）** | 侧边栏 CourseSection 点击课程 → 直接进入预习 | S4 当日预习 |
| **浏览路径（新增）** | 侧边栏「课程日历」入口 → 课程日历视图（周视图）→ 点击课程 → 进入预习 | S1、S2 |

关键设计：
- 侧边栏的 CourseSection（Today/Tomorrow/ThisWeek）**保留不变**，仍然是有课时的快速入口
- **新增**一个「课程日历」导航项（类似 Search/Recent/Starred），始终可见
- 点击「课程日历」→ ContentArea 切换为课程日历周视图
- 在周视图中点击某课程 → 进入预习空间

这样 S4 仍是两步（侧边栏点击 → 预习），S1/S2 是三步但它们本身就是探索性操作，多一步可以接受。

**脆弱性声明**：
> 两条路径并存可能导致用户困惑：「从侧边栏课程列表进入预习」和「从日历视图进入预习」是相同的目的地但不同的入口。需要确保视觉上的一致性和返回路径的正确性。

### Round 3 - Layer 1

**Proposer：**

> **Reviewer 挑战 4**：课程日历导航项的位置
>
> 关于这个挑战指向的是方案层面（信息架构），有效范围为侧边栏的导航结构。

**承认有效部分**：你说得对，如果「课程日历」作为 SidebarSection，和下方的 CourseSection 课程列表同时存在，确实会造成两个入口指向相似功能的困惑。

**方案调整**：将两者**合并**为一个统一的侧边栏区域，而不是分成两个独立的入口。

**调整后的设计**：

```
侧边栏结构：
  Search
  Recent
  Starred
  ───────────── (分割线)
  📅 课程日历 ← 始终可见的入口，点击后 ContentArea 切换为日历周视图
     ├── [Today 课程列表]    ← 有课时展开显示，快速预习入口
     ├── [Tomorrow 课程列表]
     └── [This Week 课程列表]
  ───────────── (分割线)
  ProjectTree
```

关键设计点：
- 「课程日历」是一个**可折叠的分区标题**（类似当前 ProjectTree 的交互模式），**始终可见**
- 点击分区标题 →ContentArea 切换为课程日历周视图（新的 SidebarSection: `"calendar"`）
- 分区标题下方折叠显示 Today/Tomorrow/ThisWeek 课程列表（即现有 CourseSection 的内容）
- 当日有课时，课程列表自动展开；无课时折叠但标题始终可见

> **Reviewer 挑战 5**：CourseSection 无课日行为
>
> 基于上述合并方案：

**无课日行为**：
- 「📅 课程日历」标题始终可见
- 标题右侧显示今日课程数（如 `0` 或无数字）
- 点击标题 → 进入日历周视图（用户可浏览未来课程）
- 子列表（Today/Tomorrow/ThisWeek）为空时，不显示子列表区域（保持侧边栏整洁）

**有课日行为**：
- 标题显示今日课程数（如 `3`）
- 子列表自动展开，显示 Today/Tomorrow/ThisWeek 课程
- 点击标题 → 进入日历周视图
- 点击子列表中的课程 → 直接进入预习空间（快速路径保留）

**脆弱性声明**：
> 如果用户从未导入过 ICS 课表，「课程日历」入口虽然始终可见，但进入后会是空的周视图。需要设计一个引导状态（empty state），引导用户去设置中导入课表。
